# LLM Model Comparison Dashboard - Design Document

**Date:** 2026-03-08
**Status:** Approved

## Overview

Expand the LLM Benchmarks site with model profiles, comparison dashboard, and rankings. Data sourced from OpenRouter API (pricing, latency, specs) at build-time and benchmark scores via Claude Code scraping command persisted in JSON.

## Data Architecture

### Sources

| Data | Source | Method |
|------|--------|--------|
| Pricing (input/output/cache/reasoning) | OpenRouter `/api/frontend/stats/endpoint` | Build-time fetch |
| Latency (p50-p99) | OpenRouter `/api/frontend/stats/endpoint` | Build-time fetch |
| Throughput (p50-p99) | OpenRouter `/api/frontend/stats/endpoint` | Build-time fetch |
| Specs (context window, modalities, features) | OpenRouter `/api/frontend/models` | Build-time fetch |
| Benchmark scores (SWE-bench, HumanEval, MMLU, etc.) | Official blogs, papers, LMSYS CSV | Claude Code `/scrape-benchmarks` command |

### File Structure

```
src/data/
├── models.json          # Generated by scraping (benchmark scores only)
├── providers.ts         # Static metadata (provider name, color, logo URL, OpenRouter slug)
├── benchmarks.ts        # Benchmark definitions (extracted from existing home.ts)
└── home.ts              # Existing data, unchanged
```

### OpenRouter Data (fetched at build-time, not persisted)

From `/api/frontend/stats/endpoint?permaslug={slug}&variant=standard`:
- `pricing.prompt`, `pricing.completion`, `pricing.input_cache_read`, `pricing.internal_reasoning`
- `stats.p50_latency` through `stats.p99_latency`
- `stats.p50_throughput` through `stats.p99_throughput`
- `context_length`, `max_completion_tokens`
- `input_modalities`, `output_modalities`
- `supports_reasoning`, `supports_tool_parameters`
- `is_deranked`, `status`

From `/api/frontend/models` (catalog):
- Full model list with slugs for discovery

## Pages and Components

### Pages

- **`/models/`** - Grid of all models with mini-cards, filterable by category, sortable by benchmark area
- **`/models/[slug]`** - Full model profile (radar chart, score table, pricing, specs, latency)
- **`/compare`** - Comparison dashboard (select 2+ models, radar overlay, bar chart detail)

### New Components

| Component | Purpose |
|-----------|---------|
| `ModelCard.astro` | Mini-card for grid: name, provider logo, top 3 scores, price badge |
| `ModelProfile.astro` | Full profile page content |
| `ComparisonDashboard.astro` | Model selector + charts + comparison table |
| `RadarChart.astro` | ECharts radar/spider chart wrapper |
| `BarChart.astro` | ECharts horizontal grouped bar chart wrapper |
| `RankingTable.astro` | Sortable table by category benchmark averages |
| `ModelSelector.astro` | Dropdown/chips to pick models for comparison |

### UI Flow

```
[Models Grid /models/]
  ├── click model ──► [Model Profile /models/slug]
  │                      ├── Radar chart (individual)
  │                      ├── Score table (all benchmarks)
  │                      ├── Pricing breakdown
  │                      ├── Latency percentiles
  │                      └── Specs & features
  │
  └── select 2+ models ──► [Comparison /compare]
                              ├── Radar chart (overlaid polygons)
                              ├── Click benchmark ──► Bar chart (detail)
                              └── Side-by-side comparison table
```

### Rankings

On `/models/` page, reuse existing filter pills pattern (agentic, coding, reasoning, general, multi-agent). When a category is selected, models reorder by weighted average of relevant benchmark scores for that category.

Benchmark-to-category mapping already exists in `home.ts` via `relevanceScores`.

## Charts: Apache ECharts

**Library:** Apache ECharts (via npm, tree-shaken for radar + bar only)

**Reasons:**
- Native radar chart with multi-series overlay (core requirement)
- Smooth transitions/animations when switching compared models
- Rich interactive tooltips
- Customizable themes (integrate with existing dark/light toggle)
- Chart.js would need extra plugins for equivalent radar quality

**Integration:** Client-side script tags. Data embedded in HTML by Astro at build-time, ECharts initializes on mount.

## Scraping Command

### `.claude/commands/scrape-benchmarks.md`

Claude Code command that:
1. Reads current `models.json` and `providers.ts` to know which models need data
2. For each model (or a specific model via `--model <id>`), searches official sources:
   - Provider blogs/announcements (e.g., anthropic.com/research, openai.com/index)
   - LMSYS Arena leaderboard CSV from HuggingFace
   - Official papers/technical reports
3. Extracts benchmark scores for the defined benchmark set
4. Updates `models.json` with new/changed scores
5. Sets `scrapedAt` timestamp per model

### Usage

```bash
# Scrape all models
/scrape-benchmarks

# Scrape single new model
/scrape-benchmarks --model grok-3

# Then deploy
npm run deploy
```

## Build Pipeline

```
astro build
  ├── Read src/data/models.json (benchmark scores from scraping)
  ├── Fetch OpenRouter /api/frontend/stats/endpoint for each model (pricing, latency, throughput)
  ├── Fetch OpenRouter /api/frontend/models for catalog/specs
  ├── Merge all data into unified model objects
  ├── Generate static pages with embedded data
  └── Deploy to Cloudflare Pages
```

## Design Constraints

### In Scope (v1)
- 15 models across 8 providers
- 14 benchmarks (matching existing documented benchmarks)
- Radar + bar chart comparison
- Individual model profiles
- Category-based rankings
- Dark/light theme support
- Bilingual (PT-BR/EN)
- Static site generation

### Out of Scope (v1)
- Historical score trends over time
- Real-time latency measurement (use OpenRouter's stats)
- Open-source models on third-party providers
- SSR or custom API
- User accounts or saved comparisons
